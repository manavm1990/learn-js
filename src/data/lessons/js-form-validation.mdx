# JavaScript Form Validation: Better Feedback, Same Form

## Introduction

**Icebreaker:** Think about a form you used recently. Did it help you fix mistakes while typing, or only after you clicked submit?

**Scenario:** You already know how to capture form data. Now we improve the user experience: show clear validation messages and guide the user before submit.

**Learning objectives:** You will:

- Use the browser’s built-in validation checks (`validity`, `setCustomValidity`) when needed
- Show simple, live error messages as the user types
- Enable/disable submit based on current validity
- Add a small quality-of-life feature: live character counter
- Keep code organized with small helper functions that return HTML strings

---

## Core Concept Overview

### 1. The Browser Already Knows a Lot

When an input has attributes like `required`, `minlength`, `maxlength`, or `type="email"`, the browser can check those rules for us.

Each field gives us a `validity` object.

Example checks:

- `valueMissing`
- `tooShort`
- `tooLong`
- `typeMismatch`
- `customError`

This means we do **less manual rule writing** and more clear feedback.

### 2. Native Rules First, JavaScript Second

Use HTML attributes for baseline rules.
Then use JavaScript to:

- collect errors in one place
- render readable messages
- control submit button state

### 3. Data Flow for This Lesson

Input changes → browser validity checks → our helper functions → error list in DOM → user fixes input.

### Try It 2: Gather and Render Errors

Add this to `src/practice.js`:

```js
import "./style.css";

const form = document.querySelector("#join-form");
const errorsBox = document.querySelector("#validation-errors");
const submittedBox = document.querySelector("#submitted");
const submitBtn = form.querySelector('[type="submit"]');

function ErrorItem(message) {
  return `<li class="text-red-700">${message}</li>`;
}

function gatherFieldErrors(field) {
  const validity = field.validity;
  if (validity.valid) return [];

  const errors = [];

  if (validity.valueMissing)
    errors.push(`${field.name} is required.`);

  if (validity.tooShort)
    errors.push(
      `${field.name} must be at least ${field.getAttribute("minlength")} characters.`,
    );

  if (validity.tooLong)
    errors.push(
      `${field.name} must be at most ${field.getAttribute("maxlength")} characters.`,
    );

  if (validity.typeMismatch)
    errors.push(`Please enter a valid ${field.type}.`);

  if (validity.customError)
    errors.push(field.validationMessage);

  return errors;
}

function collectErrors() {
  const fields = Array.from(form.elements).filter((field) => field.name);
  return fields.flatMap((field) => gatherFieldErrors(field));
}

function renderErrors(list) {
  errorsBox.innerHTML = list.length
    ? `<ul class="space-y-1">${list.map((item) => ErrorItem(item)).join("")}</ul>`
    : "";
}

function updateSubmitState() {
  submitBtn.disabled = collectErrors().length > 0;
}

form.addEventListener("input", () => {
  renderErrors(collectErrors());
  updateSubmitState();
});
```

**Test:** Type invalid values and verify that messages appear live.

<Accordion question="Hint: What should happen when an input becomes valid again?">
  Your code should re-run the checks every time input changes. That means old
  messages can disappear and the submit button can re-enable automatically.
</Accordion>

### Try It 3: Add One Custom Message

Let’s make one message friendlier. In `src/practice.js`, add:

```js
form.username.addEventListener("input", () => {
  // Always clear first.
  form.username.setCustomValidity("");

  // Plain-language custom rule: no spaces in username.
  if (form.username.value.includes(" "))
    form.username.setCustomValidity("Username cannot include spaces.");
});
```

**Test:** Type a space in username and verify your custom message appears.

<Accordion question="Hint: Why clear custom validity first?">
  `setCustomValidity("")` resets old custom errors. If you skip this, the
  field may stay invalid even after the user fixes the input.
</Accordion>

### Try It 4: Character Counter + Submit Handler

Still in `src/practice.js`, add:

```js
const bio = form.bio;
const bioCounter = document.querySelector("#bio-counter");
const maxChars = Number(bio.getAttribute("maxlength"));

function updateCounter() {
  const used = bio.value.length;
  const remaining = maxChars - used;
  bioCounter.textContent = `${remaining} characters remaining`;
}

bio.addEventListener("input", updateCounter);

form.addEventListener("submit", (event) => {
  event.preventDefault();

  const errors = collectErrors();
  renderErrors(errors);

  if (errors.length > 0) return;

  const data = Object.fromEntries(new FormData(form).entries());

  submittedBox.innerHTML = `
    <div class="rounded border bg-green-50 p-4">
      <h2 class="mb-2 font-semibold">Submission Received</h2>
      <pre class="rounded border bg-white p-2 text-xs">${JSON.stringify(data, null, 2)}</pre>
    </div>
  `;
});

updateCounter();
updateSubmitState();
```

**Test:**

1. Invalid form → errors show, submit blocked.
2. Valid form → JSON renders in `#submitted`.

<Accordion question="Hint: Why check again on submit?">
  Button state helps guide users, but submit is the final gate. Re-checking on
  submit protects you from missed edge cases and keeps behavior predictable.
</Accordion>

## Troubleshooting

- **Custom message never clears?** Make sure `setCustomValidity("")` runs before checks.
- **Submit always disabled?** One field is still invalid; inspect each field’s `validity` state.
- **Counter looks wrong?** Confirm `maxlength` exists in your HTML.

## Wrap-Up & Assessment

### Key Takeaways

- Let HTML attributes handle baseline validation rules.
- Use built-in browser checks (like `valueMissing`, `tooShort`, `typeMismatch`) to gather errors.
- Show helpful feedback while users type.
- Re-check on submit, even if the button is usually disabled.
- Small helper functions keep UI code readable.
